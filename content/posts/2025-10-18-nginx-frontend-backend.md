## 背景

刚开始接触前后端分离项目时，我对 Nginx 的印象比较模糊，觉得它好像就是“部署的时候顺手装一个”的工具。

后来真正把项目从本地开发走到部署阶段，才逐渐明白：

> Nginx 不是一个可有可无的配角，它解决的是“怎么把前端资源、后端服务和外部访问组织起来”的问题。

尤其在前后端分离项目里，Nginx 很常见，因为它至少能承担三类职责：

1. 托管前端构建产物
2. 作为后端服务的反向代理入口
3. 帮助前后端以统一域名或入口对外提供服务

## 如果没有 Nginx，会发生什么

先看一个最朴素的部署方式：

- 前端打包后放在某个静态服务器上
- 后端 Spring Boot 程序自己监听一个端口
- 前端直接请求后端接口地址

这种方式当然不是不能用，但很容易出现几个问题。

### 1. 访问入口不统一

前端可能是：

```text
http://example.com
```

后端接口可能是：

```text
http://example.com:8080/api/...
```

这对外暴露了服务端口，也不够整洁。

### 2. 跨域问题容易出现

如果前端和后端不在同域名同端口下，浏览器就可能触发跨域限制。

### 3. 后端直接暴露不够优雅

Spring Boot 应用更适合专注于业务处理，不一定需要直接承担对外静态资源服务和统一入口管理。

这时 Nginx 就很自然地成为中间层。

## Nginx 最重要的能力：反向代理

所谓反向代理，可以简单理解为：

- 用户访问的是 Nginx
- Nginx 再把请求转发给真正的后端服务

例如用户访问：

```text
http://example.com/api/shop/1
```

实际上是 Nginx 收到请求后，转发给后端 Spring Boot 服务：

```text
http://127.0.0.1:8080/api/shop/1
```

对用户来说，后端服务的真实地址和端口被隐藏了。

## 反向代理具体带来了什么好处

1. **统一对外入口**
2. **隐藏后端真实地址**
3. **更方便做后续扩展**
4. **更容易解决跨域问题**

## Nginx 在前端静态资源托管中的作用

前端项目打包后，本质上就是一堆静态文件：

- HTML
- CSS
- JavaScript
- 图片等资源

Nginx 很擅长提供静态资源服务，所以一个很常见的做法是：

- 把前端构建产物放到 Nginx 指定目录
- 用户访问域名时，Nginx 直接返回这些静态文件

例如：

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }
}
```

相比让 Spring Boot 顺便去处理静态页面，这种分工通常更自然：

- Nginx 负责静态内容
- Spring Boot 负责业务 API

## 一个典型的前后端分离配置思路

假设：

- 前端页面由 Nginx 托管
- 后端服务运行在 `127.0.0.1:8080`

那么 Nginx 可以这样配置接口代理：

```nginx
server {
    listen 80;
    server_name example.com;

    location / {
        root /usr/share/nginx/html;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

这样：

- 访问 `/` 时返回前端页面
- 访问 `/api/**` 时转发给后端服务

## 为什么这比前端直接连后端更好

如果前端直接写死后端地址，例如：

```javascript
axios.get("http://localhost:8080/api/shop/1");
```

在本地开发当然能跑，但到了测试或生产环境，就会变得很麻烦：

- 地址切换不方便
- 环境差异明显
- 跨域处理复杂

而通过 Nginx 代理后，前端只需要访问相对路径：

```javascript
axios.get("/api/shop/1");
```

至于请求最终被转发到哪里，由 Nginx 统一决定。

## Nginx 除了代理和静态资源，还能做什么

虽然在我的项目里主要关注的是反向代理和静态资源托管，但 Nginx 实际上还能承担很多职责，例如：

- 负载均衡
- HTTPS 终止
- 请求限流
- Gzip 压缩
- 访问日志记录
- 缓存静态资源

不过对学习实践项目来说，先把最核心的两件事理解清楚就已经很有价值了：

1. 它帮我统一了访问入口
2. 它把前端资源和后端接口组织得更合理

## 在部署层面，Nginx 为什么能提升“稳定性”

更具体一点说，我觉得它主要体现在几个方面：

1. **静态资源由 Nginx 直接处理**
2. **请求入口更可控**
3. **后端服务可以更专注**
4. **为后续扩展留下空间**

所以“稳定性”不是凭空来的，而是因为职责被拆开了。

## 开发环境和生产环境里的差异

本地开发时，很多人会用前端开发服务器自带的代理能力，比如 Vite 或 Vue CLI 的 devServer 代理。

这和 Nginx 其实解决的是同一类问题：

- 开发环境用前端工具代理
- 生产环境用 Nginx 代理

## 一些很容易忽略的配置细节

1. `try_files` 对前端路由很重要。
2. `proxy_pass` 结尾斜杠要看清楚。
3. 请求头转发别漏掉。
4. 静态资源缓存策略可以后续优化。

## 我对 Nginx 的理解变化

以前我会把 Nginx 看成一种“部署工具”，后来更觉得它像是系统入口的组织者。

它不是在替代后端，而是在做这几件非常重要的事：

- 帮前端页面找到一个稳定入口
- 帮后端接口隐藏真实地址
- 帮整个系统对外呈现为一个统一服务

## 总结

在前后端分离项目中，Nginx 最核心的价值并不神秘，主要就是三件事：

1. **托管前端静态资源**
2. **反向代理后端接口**
3. **统一系统对外访问入口**

它让前端和后端的职责划分更清楚，也让部署结构更接近真实生产环境。

这次实践让我更明确地意识到：

**很多工程问题的关键，不在于某个框架本身多强，而在于你是否给系统增加了合适的中间层，让不同部分各司其职。**
