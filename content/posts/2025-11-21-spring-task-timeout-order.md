## 背景

在订单类系统里，下单成功并不代表业务流程就结束了。很多订单在创建后还会处于“待支付”状态，而这个状态不能无限持续下去。

如果系统不处理超时未支付订单，通常会带来几个问题：

1. 订单表里会长期堆积无效订单
2. 某些业务资源可能被长时间占用
3. 后续统计、运营和库存逻辑都可能受到影响

所以“未支付订单自动关闭”并不是一个锦上添花的小功能，而是订单系统里非常常见的一项基础能力。

在项目里，我使用的是 **Spring Task 定时任务** 来实现这件事。

## 为什么需要自动关闭订单

### 1. 保持订单状态清晰

一个订单如果长时间停留在“待支付”，会让数据状态变得模糊。系统需要在某个时间点明确地把它归类为“已关闭”。

### 2. 释放业务资源

在某些场景里，待支付订单可能意味着：

- 占用了库存
- 占用了优惠资格
- 占用了某些活动名额

如果不及时释放，后续业务会受影响。

### 3. 降低无效数据对系统的干扰

超时订单如果长期不处理，后续查询、统计、运营分析都会被这些无效数据污染。

## 最简单的实现思路是什么

最直接的做法是：

1. 定时扫描数据库中的待支付订单
2. 找出创建时间已经超过阈值的记录
3. 把这些订单状态更新为“已关闭”

这也是 Spring Task 最常见的落地方式。

例如，业务规则是：

- 订单创建 15 分钟后仍未支付，就自动关闭

那么定时任务就可以每隔一段时间执行一次，筛出：

- 状态 = 待支付
- 创建时间 < 当前时间 - 15 分钟

的订单，然后批量更新。

## 为什么 Spring Task 适合做这类功能

Spring Task 的优点很明显：

1. **集成简单**
2. **适合中小规模周期任务**
3. **开发成本低**

这也是为什么很多学习项目和中小业务都会先从它开始做。

## 一个典型的实现示例

首先开启定时任务能力：

```java
@EnableScheduling
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}
```

然后定义任务：

```java
@Component
public class OrderTimeoutTask {

    @Scheduled(fixedDelay = 60000)
    public void closeTimeoutOrders() {
        LocalDateTime deadline = LocalDateTime.now().minusMinutes(15);
        orderService.closeTimeoutOrders(deadline);
    }
}
```

业务层再去查找并关闭超时订单：

```java
@Transactional
public void closeTimeoutOrders(LocalDateTime deadline) {
    List<Order> orders = orderMapper.selectList(
            Wrappers.<Order>lambdaQuery()
                    .eq(Order::getStatus, OrderStatus.PENDING_PAYMENT)
                    .lt(Order::getCreateTime, deadline)
    );

    for (Order order : orders) {
        order.setStatus(OrderStatus.CLOSED);
        orderMapper.updateById(order);
    }
}
```

## 这个任务的核心判断条件是什么

自动关闭订单时，我会特别关注两个维度：

1. **状态必须是待支付**
2. **创建时间必须超过超时阈值**

所以一条订单是否应该被关闭，通常是：

```text
status = 待支付 AND create_time < 当前时间 - 超时时间
```

## 为什么这个方案能用，但不是最终形态

Spring Task 做自动关闭非常直观，但它更像是一个“够用的起点”，不是所有订单系统的最终方案。

### 它的优点

- 上手快
- 易理解
- 适合学习项目和中小流量业务

### 它的局限

- 基于轮询扫描
- 扫描频率和精度有限
- 多实例下可能重复执行
- 订单量大时数据库压力会上升

## 轮询扫描的代价是什么

1. **存在时间误差**
2. **会不断扫数据库**
3. **数据量大时效率会下降**

所以这个方案虽然简单，但本质上是“主动扫表找任务”，而不是“订单一到时间就精准触发”。

## 多实例部署时要注意什么

如果系统部署成多个实例，而每个实例都启用了同一个定时任务，那么就可能出现重复执行。

### 常见处理方式

1. **分布式锁**
2. **调度中心统一调度**
3. **SQL 条件幂等更新**

例如更新语句里加状态条件：

```sql
update orders
set status = 'CLOSED'
where id = ?
  and status = 'PENDING_PAYMENT'
```

这样就算重复执行，也不会把其他状态误改掉。

## 自动关闭后，还要不要做补充动作

通常不只是改一个订单状态就结束。

实际业务里，订单关闭后可能还要做这些事：

- 恢复库存
- 退回优惠券资格
- 记录操作日志
- 通知前端或用户状态变化

所以“关闭订单”最好被理解成一个业务流程节点，而不是单独的一条 SQL 更新。

## 什么情况下不适合继续用 Spring Task

如果出现下面这些情况，我会觉得单纯的定时扫描开始不够用了：

1. 订单量很大
2. 对触发时间要求更精准
3. 多实例很多
4. 后续动作很多

这时可能会考虑：

- 延迟队列
- 时间轮
- 消息中间件延迟消息
- 专门的任务调度系统

## 为什么我仍然觉得这个方案值得写

因为它很真实。很多业务系统并不是一开始就有完善的延迟消息和调度中心，很多场景下，先用 Spring Task 把规则跑通，是非常务实的选择。

而且通过这个问题，我第一次比较清楚地理解了一件事：

> 很多后端功能真正的价值，不是“有没有这个接口”，而是“系统能不能在合适的时间，主动推进业务状态变化”。

## 我对这个问题的理解变化

一开始我会把它看成一个简单的定时任务功能：

- 每分钟扫一次
- 找出超时订单
- 改状态

后来再看，我觉得它背后其实涉及三个层面：

1. **业务规则**
2. **调度方式**
3. **状态推进**

也正因为这样，这个功能虽然不复杂，却很适合拿来体现后端对业务流转的理解。

## 总结

未支付订单自动关闭，本质上是在解决一个很基础但很重要的问题：

> **让待支付订单在超过有效时间后，自动从“悬而未决”变成“明确关闭”。**

Spring Task 之所以适合做这个功能，是因为它：

- 简单
- 易落地
- 足够支撑中小规模业务

但它的边界也很清楚：

- 依赖轮询
- 精度一般
- 多实例下要额外处理
- 数据量大时会面临性能问题

对我来说，这次实践最大的收获是：

**后端很多看似普通的功能，真正的难点不是写出一段任务代码，而是把业务状态变化的规则和边界想清楚。**
