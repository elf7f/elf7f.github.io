## 背景

在做带有秒杀优惠券功能的项目时，我最先遇到的不是“怎么把功能写出来”，而是“并发一上来，这个功能还能不能成立”。

秒杀业务看起来很简单：用户点击抢购，系统判断是否还有库存，如果有就创建订单并扣减库存。但只要并发量上来，下面几个问题就会一起出现：

1. 多个请求同时读到库存还有剩余，导致**超卖**
2. 同一个用户快速重复点击，导致**一人多单**
3. 如果每次都直接访问数据库，会让数据库在高峰期承受很大压力

也就是说，秒杀的难点不在于“下单”本身，而在于**如何在高并发下把校验和扣减做成一个原子动作**。

我在项目里采用的方案是：**把库存判断和一人一单校验前移到 Redis，并通过 Lua 脚本保证整段逻辑的原子性**。

## 为什么不能直接在数据库里做

最直观的写法通常是下面这样：

1. 查询优惠券库存
2. 判断库存是否大于 0
3. 查询当前用户是否已经下过单
4. 如果没下过单，则扣减库存
5. 创建订单

业务逻辑看起来没有问题，但在高并发下会出现经典竞态条件。

### 问题一：超卖

假设库存只剩 1 件，有两个请求同时进来：

- 请求 A 读到库存 = 1
- 请求 B 也读到库存 = 1
- 两个请求都认为自己可以下单
- 最后都执行扣减库存和创建订单

结果就是**库存被卖成了 2 单**。

### 问题二：一人多单

同一个用户如果短时间内快速点击两次，或者客户端重试了请求，也会出现类似问题：

- 第一次请求还没来得及写订单
- 第二次请求已经完成了“未下单”的判断
- 最终同一个用户生成两笔订单

### 问题三：数据库扛不住峰值

秒杀的特点是**高峰非常集中**。哪怕最终只有极少数请求成功，所有失败请求依然可能先把数据库打满。

所以我的思路是：**让大多数无效请求在进入数据库之前就被拦下**。

## 为什么选择 Redis

Redis 很适合做秒杀前置校验，主要有三个原因。

### 1. 内存访问快

相比数据库，Redis 的响应速度更快，更适合承接高并发读写。

### 2. 天然适合保存秒杀态数据

例如：

- 某个优惠券当前剩余库存
- 哪些用户已经抢到过

这些数据都适合放在 Redis 中。

### 3. 能借助 Lua 实现原子执行

如果只是把“库存”放进 Redis，但校验步骤仍然拆成多次命令执行，问题并不会真正消失。因为多条命令之间仍然可能被并发请求穿插。

Redis + Lua 的关键价值在于：

> 把“判断库存”“判断用户是否已下单”“扣减库存”“记录用户购买资格”放进同一个脚本中一次执行。

这样可以把原本分散的多步逻辑，变成一个不可分割的原子操作。

## 核心思路：把资格校验放到 Redis 前面做

整个处理链路可以分成两个阶段：

### 第一阶段：资格校验与库存预扣减

在 Redis 中完成：

1. 判断库存是否大于 0
2. 判断当前用户是否已经下过单
3. 如果通过：
   - 扣减 Redis 中的库存
   - 记录该用户已抢购

这个阶段只做“能不能抢”的判断，不直接写数据库订单。

### 第二阶段：异步创建正式订单

通过消息队列或异步消费者，把通过资格校验的请求继续往后处理：

1. 写数据库订单
2. 持久化真实库存变更
3. 执行后续业务逻辑

这样的好处是：

- 主链路非常短
- 大部分失败请求不会进入数据库
- 高峰期系统更稳定

## Lua 脚本为什么能解决原子性问题

Lua 脚本在 Redis 中执行时，会被当作一个整体处理。也就是说，脚本里的多个命令在执行期间不会被其他客户端命令插入。

一个典型脚本逻辑大概如下：

```lua
local stockKey = KEYS[1]
local orderKey = KEYS[2]
local userId = ARGV[1]

if (tonumber(redis.call('get', stockKey)) <= 0) then
    return 1
end

if (redis.call('sismember', orderKey, userId) == 1) then
    return 2
end

redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)

return 0
```

返回值约定可以是：

- `0`：允许抢购
- `1`：库存不足
- `2`：重复下单

Java 侧只需要根据返回值决定后续流程。

```java
Long result = stringRedisTemplate.execute(
        SECKILL_SCRIPT,
        Arrays.asList(stockKey, orderKey),
        userId.toString()
);

if (result == null) {
    throw new IllegalStateException("Lua script execute failed");
}
if (result == 1L) {
    return Result.fail("库存不足");
}
if (result == 2L) {
    return Result.fail("不能重复下单");
}
```

这种写法的关键不在于代码本身，而在于**把最容易出并发问题的部分收缩到 Redis 的原子脚本里执行**。

## Redis 中应该怎么设计 key

为了让脚本逻辑清晰，我更倾向于把数据拆成两类 key。

### 1. 库存 key

```text
seckill:stock:{voucherId}
```

### 2. 用户资格集合 key

```text
seckill:order:{voucherId}
```

值类型用 `Set` 更合适，用来记录已经成功抢购过的用户 id。

这样设计的好处是：

- 判断某用户是否已购买可以直接用 `SISMEMBER`
- 插入购买记录可以直接用 `SADD`
- 数据语义很明确，便于排查

## 这个方案为什么叫“库存预扣减”

因为 Redis 中的扣减，并不是最终意义上的“订单成功完成”，而是先把抢购资格锁定下来。

也就是说：

- **Redis 扣减成功**：表示该请求拿到了继续往后执行的资格
- **数据库订单落库成功**：才代表业务最终成立

所以 Redis 里的扣减，本质上是**预扣减**。

这一步非常关键，因为它可以在高峰期快速筛掉大量失败请求。但也带来一个必须正视的问题：

> 如果 Redis 预扣减成功了，后面的订单创建却失败了怎么办？

## 落地时必须考虑的一致性问题

高并发方案往往不是“做完原子扣减就结束了”，真正难的是后续一致性。

### 场景一：Redis 扣减成功，但数据库下单失败

比如：

- 消费者异常
- 数据库写入失败
- 网络抖动导致后续流程未完成

这时就会出现 Redis 认为库存少了 1，但数据库里并没有对应订单。

### 可以怎么处理

#### 做法一：失败补偿

如果数据库订单创建失败，则：

- 回滚 Redis 中的库存
- 删除用户购买标记

这是最直观的思路，但实现时要非常谨慎，避免补偿重复执行。

#### 做法二：用可靠消息最终一致性

通过消息队列、消费确认、重试机制等方式，让“预扣减成功的请求”最终尽量落成订单。

如果多次重试后仍然失败，再触发补偿逻辑。

#### 做法三：把 Redis 当作流量闸门，而不是最终事实来源

也就是承认：

- Redis 负责承接高并发流量
- MySQL 里的订单和库存才是最终业务真相

这种设计在工程上更现实，因为缓存层和消息层本来就更适合做削峰和过滤，而不是做唯一真相源。

## 这个方案相比数据库锁，有什么优势

很多人第一次做秒杀时，会想到用数据库事务、乐观锁或悲观锁控制并发。它们不是不能用，而是有场景差异。

### Redis + Lua 的优势

1. **前置拦截能力更强**：大量无效请求在 Redis 层就被挡掉，不会压到数据库。
2. **延迟更低**：内存操作比数据库锁竞争更轻，主链路更短。
3. **更适合和异步化链路组合**：资格校验和正式下单解耦后，系统吞吐更容易做上去。

### 数据库方案的优势

1. **强一致性更直接**
2. **业务真相只有一份**

所以我现在的理解是：

> Redis + Lua 更适合做高并发入口控制，数据库事务更适合做最终状态落地。

两者不是互斥关系，而是分工不同。

## 一个更完整的秒杀处理流程

1. 用户发起秒杀请求
2. 通过 Redis + Lua 判断库存和一人一单
3. 如果失败，立即返回
4. 如果成功，生成请求标识并写入消息队列
5. 后端异步消费者创建数据库订单
6. 订单创建成功后，业务完成
7. 如果失败，进入重试或补偿

```text
用户请求
   ↓
Redis + Lua 原子校验
   ↓
成功 → 写入 MQ → 异步下单 → 落库成功
   ↓
失败 → 返回“库存不足”或“重复下单”
```

## 实践中我会关注的几个细节

1. 用户 id 一定要可靠，一人一单依赖用户唯一标识。
2. key 的生命周期要设计好，活动结束后要考虑过期或清理。
3. 返回值要足够明确，排查问题时会方便很多。
4. 不要把 Redis 结果当最终订单结果，拿到秒杀资格不等于订单已创建成功。

## 这种方案的局限

1. 引入了缓存与数据库双写一致性问题
2. 更依赖后续异步链路稳定性
3. 业务复杂后脚本也会变复杂

所以我现在会把 Lua 脚本控制在一个边界里：

> 只做高并发下必须原子化的那部分逻辑，不承担整个业务系统的全部责任。

## 总结

秒杀场景的关键不是“写一个下单接口”，而是**在极短时间内筛掉无效请求，并保证核心规则不被并发破坏**。

我之所以选择 Redis + Lua，是因为它非常适合解决两个最核心的问题：

- 库存不能超卖
- 用户不能重复抢购

而且这两个判断必须在高并发下以原子方式完成。

对我来说，这次实践最大的收获不是“学会了 Lua 语法”，而是第一次比较清楚地理解了：

**高并发系统里，真正重要的不是把业务写得更复杂，而是把核心约束尽量前置，并让它们在最短链路上完成。**
