## 背景

在刚开始做缓存优化时，我会很自然地把 Redis 当成“唯一答案”。

思路也很直接：

- 热点数据放 Redis
- 服务实例都从 Redis 读
- Redis 没有再查数据库

这个方案已经比直查数据库好很多了，但真正把请求量拉起来之后，我慢慢发现一个问题：

> **即使已经用了 Redis，热点数据仍然会不断地穿过网络去访问它。**

如果某些数据非常热，比如：

- 店铺详情
- 活动信息
- 热门商品基础信息

那么一次请求哪怕只访问 Redis，也依然要经过网络调用、序列化与反序列化。单次看不算重，但在高频热点场景下，Redis 本身也会成为压力中心。

所以后来我更理解为什么很多系统会做二级缓存：**本地缓存 + 分布式缓存**。

在我的项目里，我使用的是 **Caffeine + Redis**。

## 先回答一个问题：为什么用了 Redis 还要本地缓存

答案是：**快是相对的**。

Redis 比数据库快得多，但它依然不是“零成本访问”。

### Redis 的成本主要来自哪里

1. **网络开销**：应用访问 Redis 仍然是一次远程调用。
2. **序列化与反序列化**
3. **Redis 本身也有吞吐上限**

而本地缓存的优势在于：

- 数据在当前 JVM 进程内
- 不需要走网络
- 读取更快
- 能进一步降低 Redis 请求量

所以本地缓存不是为了替代 Redis，而是为了**继续拦截更高频的热点访问**。

## 二级缓存的典型访问路径

```text
请求 → Caffeine（本地缓存）→ Redis（分布式缓存）→ MySQL（数据库）
```

也就是说：

1. 优先查 Caffeine
2. 本地没有，再查 Redis
3. Redis 没有，再查数据库
4. 查到后，回填 Redis 和 Caffeine

这样设计后，最热的数据通常只会停留在应用进程内，不会反复穿透到 Redis。

## Caffeine 为什么适合作本地缓存

如果只是做应用内热点数据缓存，Caffeine 是一个很合适的选择。

我喜欢它，主要是因为下面几点：

1. **够轻量**
2. **性能好**
3. **支持灵活的淘汰策略**
4. **和 Spring Boot 项目结合方便**

一个简单的初始化方式大概是：

```java
Cache<Long, Shop> localCache = Caffeine.newBuilder()
        .initialCapacity(100)
        .maximumSize(10_000)
        .expireAfterWrite(10, TimeUnit.MINUTES)
        .build();
```

## 二级缓存到底解决了什么问题

### 1. 进一步降低 Redis 压力

如果某个热点数据在短时间内被反复读取，本地缓存会把这些请求直接截住，Redis 的读流量自然下降。

### 2. 降低接口响应延迟

本地内存读取几乎总是比网络访问 Redis 更快，尤其是高频热点接口，收益会很明显。

### 3. 提升系统的局部抗波动能力

如果 Redis 短时间抖动、网络波动或者延迟升高，应用层还可以依靠本地缓存顶住一部分流量。

## 典型的查询流程应该怎么写

```java
public Shop queryShop(Long shopId) {
    Shop shop = localCache.getIfPresent(shopId);
    if (shop != null) {
        return shop;
    }

    String key = "cache:shop:" + shopId;
    String json = stringRedisTemplate.opsForValue().get(key);
    if (StrUtil.isNotBlank(json)) {
        shop = JSONUtil.toBean(json, Shop.class);
        localCache.put(shopId, shop);
        return shop;
    }

    shop = shopMapper.selectById(shopId);
    if (shop == null) {
        return null;
    }

    stringRedisTemplate.opsForValue().set(key, JSONUtil.toJsonStr(shop), 30, TimeUnit.MINUTES);
    localCache.put(shopId, shop);

    return shop;
}
```

这个流程不复杂，但它体现了一件事：

> 热点数据应该尽量被拦在离业务代码最近的地方。

## 本地缓存和 Redis 的职责分工

### 本地缓存适合做什么

- 承接当前实例的超高频热点读
- 缩短读取路径
- 减少网络调用
- 提升单实例响应速度

### Redis 适合做什么

- 在多实例之间共享缓存
- 承担分布式层面的统一缓存
- 减少数据库访问
- 作为本地缓存未命中的下一层兜底

### 数据库适合做什么

- 保存最终业务真相
- 在缓存失效时提供源数据

所以二级缓存不是简单叠两层，而是：

- **Caffeine 负责最近、最热、最本地的访问**
- **Redis 负责跨实例共享的缓存状态**
- **数据库负责最终数据源**

## 二级缓存最现实的问题：一致性

二级缓存最吸引人的地方是性能，但最容易出问题的地方也是一致性。

因为现在数据可能同时存在三处：

- 本地缓存
- Redis
- 数据库

一旦数据库中的数据更新了，如果缓存没有同步更新，就会出现脏数据。

### 最常见的问题

假设店铺信息被修改：

1. 数据库更新成功
2. Redis 还没删
3. 某个应用实例的本地缓存也还在

结果就是不同实例可能同时读到不同版本的数据。

## 常见的更新策略

### 1. 旁路缓存模式

- 先更新数据库
- 再删除 Redis
- 同时删除本地缓存

```java
@Transactional
public void updateShop(Shop shop) {
    shopMapper.updateById(shop);

    String redisKey = "cache:shop:" + shop.getId();
    stringRedisTemplate.delete(redisKey);

    localCache.invalidate(shop.getId());
}
```

这种做法的优点是简单，缺点是如果部署了多个实例，**当前实例删掉本地缓存并不代表其他实例也删掉了**。

## 多实例下本地缓存一致性怎么处理

如果服务只有一个实例，那删本地缓存就够了。但在多实例部署下，每个实例都有自己的一份 Caffeine 缓存。

### 常见解决思路

1. **缩短本地缓存 TTL**
2. **更新时广播失效消息**
3. **只对“允许短暂不一致”的数据使用本地缓存**

如果是店铺名称、商品描述、推荐列表等数据，几秒到几十秒的不一致往往可以接受；如果是库存、余额、支付状态，就不适合这么做。

## 什么样的数据适合放进本地缓存

比较适合的通常是：

- 高频读
- 修改少
- 允许短暂旧数据
- 单条数据不大
- 热点明显

例如：

- 商铺详情
- 商品基础信息
- 分类列表
- 配置信息

而不太适合的是：

- 实时库存
- 支付状态
- 账户余额
- 高频写数据

## 二级缓存和“缓存雪崩/击穿”是一个问题吗

不是一回事，但会互相影响。

### 二级缓存主要解决的是

- 读性能
- Redis 压力
- 热点请求路径缩短

### 缓存击穿主要解决的是

- 热点 Key 过期瞬间数据库被打爆

现实里常常会一起做：

- 用 Redis 处理分布式缓存
- 用 Caffeine 进一步承接本地热点
- 对 Redis 热点 key 再做逻辑过期或互斥锁防击穿

## 二级缓存的代价是什么

1. **一致性更难处理**
2. **内存占用会上升**
3. **问题排查会更麻烦**
4. **并不是所有业务都值得做**

如果业务流量不高，或者数据本身不热，二级缓存带来的收益可能不如引入的维护成本大。

## 我会怎么用这套方案

如果是个人项目或中小规模服务，我更倾向于把二级缓存用在下面这些地方：

- 热点店铺详情
- 热门商品基础信息
- 变化不频繁的字典配置
- 首页列表类数据

并且遵守两个原则：

1. **只给真正热点的数据上本地缓存**
2. **接受有限度的不一致**

## 总结

Caffeine + Redis 二级缓存的本质，不是“多叠一层就更高级”，而是对缓存链路做了更细的分工：

- **Caffeine** 负责拦截当前实例的超高频访问
- **Redis** 负责跨实例共享缓存
- **MySQL** 负责保存最终数据源

这样设计最大的好处是：

- 热点请求更快
- Redis 压力更小
- 系统对高频读流量更从容

但它真正的难点在于一致性。只有当数据本身适合缓存、业务接受短暂旧值、并且你愿意为此承担一些复杂度时，这套方案才真正值得上。

这次实践让我更清楚地意识到：

**性能优化从来不是无条件加技术，而是明确每一层在系统里扮演什么角色。**
