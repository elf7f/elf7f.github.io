## 背景

做秒杀功能时，我很快发现一个问题：即使已经用 Redis + Lua 把“库存校验”和“一人一单判断”前置了，系统仍然不适合把**正式创建订单**这件事放在主请求链路里同步执行。

原因很简单。秒杀和普通下单的区别，不只是“流量大”，而是：

- 流量在极短时间内瞬间涌入
- 绝大多数请求最终都会失败
- 少量成功请求需要继续完成真正的订单处理

如果把所有成功请求都同步落库、同步执行业务逻辑，那么接口的平均耗时、数据库压力、事务冲突都会迅速放大。

所以后来我更倾向于把秒杀拆成两个阶段：

1. **快速判断是否有资格**
2. **异步完成正式订单创建**

这也是我在项目里引入 Kafka 的原因。

## 为什么同步下单链路很容易扛不住

先看一个最直观的同步流程：

1. 用户请求进入接口
2. 校验秒杀资格
3. 扣减库存
4. 创建订单
5. 写数据库
6. 返回成功

在普通业务里，这样做没有问题。但在秒杀里，这条链路往往有几个明显缺点。

### 1. 主链路太长

秒杀接口最怕的不是逻辑复杂，而是**高峰期每个请求都停留太久**。

只要主链路里还要做数据库事务、订单表写入、库存表更新以及后续业务，接口就很容易被拖慢。

### 2. 数据库容易成为瓶颈

秒杀本质是“前面来得很快，后面处理能力有限”。哪怕只有几千个成功资格请求，如果都要在同一时间写数据库，也会造成数据库连接池紧张、锁竞争上升、事务执行时间拉长。

### 3. 系统抗波动能力差

同步链路里只要数据库短暂抖动、线程池繁忙、下游服务慢一点，整个接口响应就会立刻受影响。

## 异步下单到底在解什么问题

我现在更愿意把秒杀看成一个“先拿资格，再补全结果”的过程。

也就是说，用户请求来到接口时，系统最优先做的不是“立刻把订单完整写完”，而是：

> 尽快判断这个请求有没有资格进入后续处理。

如果有资格，就把它放进一个可靠的异步处理链路里慢慢消费。

Kafka 在这个过程中扮演的角色非常明确：

- **接住高峰流量**
- **缓冲请求压力**
- **把接入层和订单处理层解耦**

这就是典型的削峰填谷思路。

## 一个更合理的秒杀链路

### 第一步：用户发起秒杀请求

接口收到请求后，先不急着创建数据库订单。

### 第二步：Redis + Lua 做资格校验

在 Redis 中原子完成库存是否充足、用户是否已经下过单。如果失败，立即返回结果。

### 第三步：把通过资格校验的请求写入 Kafka

消息内容通常至少需要包含：

- 用户 id
- 优惠券 id
- 请求时间
- 唯一请求标识

### 第四步：消费者异步创建订单

消费者从 Kafka 中取出消息，再执行真正的订单落库和业务处理。

### 第五步：结果最终落地

订单创建成功后，系统再认为这次秒杀真正完成。

用一句话概括就是：

> 同步阶段只负责“能不能抢”，异步阶段负责“怎么把订单做完”。

## Kafka 为什么适合做这件事

从秒杀这种“短时间写入集中”的场景来看，Kafka 至少有几个很匹配的特点：

1. **吞吐高**：适合承接短时间内大量消息写入。
2. **天然解耦**：秒杀接口只负责“发消息”，不直接承担全部订单逻辑。
3. **更容易做削峰**：高峰期请求先进入 Kafka，消费者按系统可承受的速度往后处理。
4. **有利于扩展消费者能力**：可以通过调整消费者并发度、消费分区等方式扩展处理能力。

## 消息里应该放什么

异步下单不是把一个“空消息”扔进队列就完了。消息体至少包含：

```json
{
  "orderId": "189320001",
  "userId": 10086,
  "voucherId": 20001,
  "createTime": "2026-03-06T09:30:00",
  "requestId": "seckill-10086-20001-1741224600"
}
```

其中 `orderId` 和 `requestId` 对幂等和排查都很重要。

## 异步之后，接口应该返回什么

如果接口在资格校验通过后就直接返回“抢购成功”，而后面的异步订单创建又失败了，那么语义上会不严谨。

所以更合理的做法通常有两种：

1. 返回“已受理”或“正在处理中”
2. 返回抢购资格成功，但明确订单结果以最终落库为准

本质上，异步系统里**接口成功 ≠ 业务最终成功**。

## 异步方案最大的收益是什么

1. **主请求链路变短**：写消息后即可返回，接口整体耗时更稳定。
2. **数据库压力更可控**：数据库面对的是被削峰后的消费流量。
3. **系统分层更清楚**：接入层负责校验和接单，消费层负责订单创建。
4. **更容易扩展和调优**：可观察消费堆积、失败率和延迟。

## 异步化之后，最容易踩的坑是什么

### 1. 消息发出去了，但订单没创建成功

这时要考虑：

- 消费是否失败
- 是否需要重试
- 最终失败后怎么补偿

### 2. 消息可能重复消费

所以消费者必须做好幂等。可以用：

- 订单号唯一约束
- 请求 id 去重
- 用户 id + 优惠券 id 唯一索引

### 3. Redis 预扣减和数据库最终结果不一致

如果 Redis 阶段已经扣了库存并标记用户资格，但异步消费者最终失败，就要考虑补偿或重试。

### 4. 消费速度跟不上生产速度

高峰期如果 Kafka 里消息快速堆积，而消费者处理能力不足，就会造成延迟升高。

## 消费者为什么必须做幂等

只要引入消息队列，就必须假设**重复消息是可能发生的**。

```java
@Transactional
public void createVoucherOrder(SeckillOrderMessage message) {
    Long userId = message.getUserId();
    Long voucherId = message.getVoucherId();

    int count = voucherOrderMapper.selectCount(
            Wrappers.<VoucherOrder>lambdaQuery()
                    .eq(VoucherOrder::getUserId, userId)
                    .eq(VoucherOrder::getVoucherId, voucherId)
    );

    if (count > 0) {
        return;
    }

    VoucherOrder order = new VoucherOrder();
    order.setId(message.getOrderId());
    order.setUserId(userId);
    order.setVoucherId(voucherId);
    voucherOrderMapper.insert(order);
}
```

真正稳妥的方案还应该有数据库唯一索引兜底。

## 什么时候不一定需要 Kafka

不是所有“异步化”都必须上 Kafka。业务规模不大时，也可以考虑：

- 内存队列
- Redis Stream
- RabbitMQ
- 阻塞队列 + 单机消费

核心问题不是“哪个更高级”，而是：

- 当前流量规模是多少
- 是否需要高吞吐写入
- 团队能不能维护这套基础设施

## 我对“削峰”的理解

削峰真正解决的是：

> **让系统各层只面对自己能承受的流量形态。**

前端用户流量通常是突发的，而数据库和业务服务更适合处理平滑、可控的写入压力。

Kafka 的存在，就是在中间加了一层缓冲带：

- 前面快进来
- 后面稳处理

## 总结

秒杀系统里的异步下单，本质上不是“把同步代码改成异步代码”这么简单，而是一次职责拆分：

- **Redis + Lua** 负责快速判断资格
- **Kafka** 负责承接瞬时流量并做削峰
- **消费者** 负责真正落地订单
- **数据库** 负责保存最终业务事实

这样做最大的价值，不只是吞吐更高，而是让整个系统在高峰流量下更稳定、更可控。

这次实践让我更清楚地意识到：

**高并发系统的设计重点，不是让每个请求都立刻做完所有事，而是先做最必要的判断，再用合适的机制把后续处理平滑地接下去。**
