## 为什么我会关注“结构化输出”

我准备做一个 `prompt-polisher` 项目，它的目标不是简单地“把一句话润色得更好看”，而是帮助用户把原本模糊、零散、信息不足的问题，整理成更适合提交给大模型的 Prompt。

一开始我想过最直接的实现方式：  
用户输入一句话，模型输出一句优化后的话。

但很快我就发现，这样的结果虽然看起来简单，却不够好用。

原因在于，用户真正需要的往往不只是“改写后的句子”，而是：

- 我的原问题到底差在哪里
- 我还缺哪些关键信息
- 改写后的版本为什么更好
- 有没有一个可以直接复制给模型的最终 Prompt

如果只返回一句“优化后的结果”，这个工具就会变成一个很普通的改写器，而不是一个真正帮助用户提高提问质量的产品。

所以我决定从一开始就把输出做成 **结构化结果**。

---

## 为什么不能只输出一段润色后的文本

如果只给用户返回一段最终文本，会有几个明显问题。

### 第一，用户不知道自己原问题的问题在哪里

例如用户输入：

> 帮我写简历

如果系统只返回：

> 请根据我的教育背景、项目经历和求职方向，帮我优化一份适合 Java 后端开发实习岗位的简历。

虽然这句话更完整了，但用户并不知道：

- 为什么原问题太模糊
- 缺了哪些信息
- 下次该怎么提问更好

### 第二，用户不知道还需要补充什么

很多问题之所以回答不理想，并不是模型不够强，而是输入缺少上下文。

例如：

- 没说目标岗位
- 没说当前水平
- 没说预算
- 没说输出格式

如果不把这些缺失信息显式指出来，用户还是难以真正改善提问习惯。

### 第三，工具价值不够明显

“改写一句话”这个能力很容易被替代。  
但“帮助用户理解问题缺陷，并生成可直接使用的 Prompt”会更像一个独立产品。

所以我认为，`prompt-polisher` 的核心竞争力不在“润色”，而在 **结构化反馈**。

---

## 我想让这个工具输出什么

在思考之后，我把输出固定成四个部分：

1. 原始问题分析
2. 建议补充的信息
3. 优化后的问题
4. 最终可复制 Prompt

这个结构看起来不复杂，但已经能覆盖用户从“意识到问题”到“拿到可用结果”的完整链路。

---

## 第一部分：原始问题分析

这一部分的目标是：

**告诉用户，他原来的问题哪里不够清楚。**

例如用户输入：

> 帮我改代码

系统可以分析出：

- 没说明使用的语言和框架
- 没提供代码片段或报错信息
- 没说明希望修改成什么效果

这一部分的价值在于，它不是直接给答案，而是先帮助用户“看见问题”。

我觉得这一步很重要，因为很多人和模型交流时，最大的问题并不是不会说话，而是意识不到自己提供的信息不足。

所以在输出上，我会把这部分设计成一个字符串列表，而不是一整段长文。例如：

```json
{
  "analysis": [
    "原问题没有说明编程语言和框架",
    "没有提供报错信息或代码片段",
    "没有说明期望得到的修改结果"
  ]
}
```

这样做有两个好处：

- 前端展示更清晰
- 用户更容易逐条理解

---

## 第二部分：建议补充的信息

仅仅指出问题还不够，还需要告诉用户：

**接下来应该补什么。**

这部分和上一部分相关，但不完全一样。

例如上面那个例子，建议补充的信息可以是：

- 代码片段
- 报错信息
- 使用的语言或框架
- 你希望达到的结果

我把这一部分单独拆出来，是因为它更偏“行动建议”。

前一部分是在说：

> 你的问题有哪些不足

这一部分是在说：

> 你现在应该补哪些信息

对于用户来说，这比抽象地说“你描述得不够详细”要具体得多。

在数据结构上，我也会把它设计成数组：

```json
{
  "missingInfo": [
    "代码片段或关键报错信息",
    "使用的编程语言或框架",
    "你希望实现的目标效果"
  ]
}
```

---

## 第三部分：优化后的问题

这一部分才是传统意义上的“改写结果”。

例如把：

> 帮我改代码

改成：

> 请帮我分析并修改这段 Java 代码中的问题。我使用的是 Spring Boot，当前遇到的报错是……，希望你指出错误原因，并给出修改后的代码和解释。

这一部分的作用是：

- 给用户一个更清晰的问法
- 帮助用户理解“更好的提问应该长什么样”

我希望它保留用户原始意图，但补齐必要上下文，并让表达方式更明确。

它和第四部分的区别在于：

- 第三部分更像“改写后的自然问题”
- 第四部分更像“可直接提交给模型的完整指令”

---

## 第四部分：最终可复制 Prompt

这是整个工具里最偏“产品输出”的部分。

有些用户并不在意中间分析过程，他们最关心的是：

**最后我能不能拿到一段直接复制给模型的 Prompt。**

所以第四部分应该更完整、更明确，也可以稍微带一点“提示词工程”的味道。

例如：

> 你是一名资深 Java 开发工程师。请根据我提供的代码、报错信息和目标要求，分析当前问题的原因，并给出修改建议与完整代码示例。请解释修改思路，并指出可能的边界情况。

这类文本通常会比第三部分更正式，也更适合直接提交给模型。

我把这一部分单独设计出来，是为了满足“直接可用”的需求。

---

## 为什么我选择这四段结构

这个结构不是随便拆的，而是对应用户使用这个工具时的四个真实需求：

### 1. 我的问题哪里不对
对应：原始问题分析

### 2. 我还缺什么信息
对应：建议补充的信息

### 3. 更清晰的表达应该是什么样
对应：优化后的问题

### 4. 我现在能直接复制什么去问模型
对应：最终可复制 Prompt

这样设计之后，整个产品就不只是“输出一句更好的话”，而是一个完整的改写与引导流程。

---

## 为什么返回 JSON 而不是纯文本

从工程实现角度，我也更倾向于让模型返回 JSON，而不是返回自由文本。

原因很直接。

### 第一，前端更容易展示

如果返回 JSON，前端可以把每个字段单独放进卡片中，例如：

- 问题分析卡片
- 缺失信息卡片
- 优化结果卡片
- 最终 Prompt 卡片

这样结构清晰，用户体验也更好。

### 第二，更容易校验

如果模型返回的是：

- `analysis`
- `missingInfo`
- `optimizedPrompt`
- `finalPrompt`

这些固定字段，后端就更容易判断结果是否完整。

### 第三，后续扩展更方便

后续如果要增加：

- 置信度
- 场景标签
- 风格选项
- 多版本改写结果

JSON 会更容易扩展，而纯文本很难稳定解析。

所以在后端设计上，我会要求模型严格输出类似这样的结构：

```json
{
  "analysis": [],
  "missingInfo": [],
  "optimizedPrompt": "",
  "finalPrompt": ""
}
```

---

## 不同场景下，结构相同但内容不同

虽然输出结构固定，但不同场景下关注点是不一样的。

这也是我为什么要在项目里加入 `scene` 这个字段。

### 编程场景

更关注：

- 编程语言
- 框架
- 报错信息
- 输入输出
- 期望结果

### 学习场景

更关注：

- 当前水平
- 学习目标
- 时间安排
- 输出偏好

### 写作场景

更关注：

- 文章类型
- 目标读者
- 语气风格
- 长度要求

### 求职场景

更关注：

- 岗位方向
- 当前背景
- 目标城市
- 输出用途（简历、面试、自我介绍）

所以我的设计思路是：

- 输出结构固定
- 场景规则变化
- 模型根据场景填充内容

这样产品既统一，又有针对性。

---

## 前端应该怎么展示这个结构

如果后端已经返回结构化 JSON，前端其实很容易做展示。

我会把页面结果区域拆成四个卡片：

### 卡片一：问题分析
展示 `analysis`

### 卡片二：建议补充
展示 `missingInfo`

### 卡片三：优化后的问题
展示 `optimizedPrompt`

### 卡片四：最终 Prompt
展示 `finalPrompt`，并增加一个 Copy 按钮

这样用户一眼就能理解每一部分的作用。

而且这种卡片式展示很适合做成一个简洁工具页，不需要复杂 UI。

---

## 这种结构化输出的好处

当我把整个设计梳理清楚后，我觉得这个结构有几个很明显的优势。

### 1. 用户更容易理解问题本身

不是只拿到结果，而是理解“为什么要这样改”。

### 2. 工具更像产品，而不是简单包装 AI 接口

如果只是一句改写文本，这个项目会显得很薄。  
但结构化输出让它更像一个有完整交互逻辑的小产品。

### 3. 更适合前后端协作

后端返回标准 JSON，前端按模块渲染，工程边界很清晰。

### 4. 更方便后续迭代

以后可以很自然地加入：

- 多版本结果
- 风格选项
- 历史记录
- 对比视图

---

## 我认为这个项目最重要的不是 API，而是输出设计

做这类 AI 工具时，很容易把注意力放在：

- 调哪个模型
- 用哪个接口
- 前端怎么调 API

但我觉得 `prompt-polisher` 最核心的部分其实不是这些，而是：

**最后到底要输出什么。**

如果输出设计得不清楚，这个工具就会变得很普通。  
相反，只要输出结构合理，哪怕第一版 UI 很简单，它也会显得有思考、有产品感。

---

## 小结

如果让我总结 `prompt-polisher` 的结构化输出设计，我会概括成下面几点：

- 不只返回一句润色后的文本
- 把结果拆成“分析、补充建议、优化结果、最终 Prompt”四部分
- 用 JSON 作为统一输出格式
- 不同场景复用同一结构，只调整关注重点
- 前端用卡片化方式展示，提升可读性和产品感

对我来说，这种结构化输出的设计，才是 `prompt-polisher` 这个项目真正有意思的地方。
